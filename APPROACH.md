# Approach

## Design

The service follows the assignment API exactly: `GET /health` returns `{"status": "ok"}` and `POST /chat` accepts a full stateless conversation history. The API layer is intentionally small; it validates request/response shape with Pydantic and delegates recommendation logic to `recommender.py`.

The existing catalog work is preserved. `DB.py` is the manual vector database creation path and should be run after installing requirements and before starting `uvicorn`. Retrieval is handled through `retrieval.py`, which queries the Chroma database created from the fixed `shl_product_catalog.json` included in the repository. The originally provided catalog JSON had a formatting issue around line 4795, so the GitHub version keeps the manually corrected file as the source of truth. Final recommendations are validated against this catalog so returned URLs and names remain catalog-backed.

## Conversation Handling

The recommender reads all user turns on every request and lets Groq decide the next conversational move. When catalog grounding is needed, the RAG pipeline calls `Search_indocs`, which searches the existing Chroma vector database created from the SHL catalog.

- clarify vague queries before recommending;
- recommend 1 to 10 catalog assessments once enough context exists;
- refine recommendations when users add or remove constraints;
- compare catalog products using retrieved catalog context;
- refuse off-topic, legal, and prompt-injection requests.

This keeps the service fast enough for the 30-second evaluator timeout and avoids hidden server-side memory.

The evaluator limits are handled explicitly:

- `MAX_TURNS` is set to 15 messages. This follows the sample-conversation interpretation where one user+assistant exchange is treated as one turn, while still preventing very long histories;
- the full recommendation pipeline is bounded to 25 seconds so the API can respond before the evaluator's 30-second timeout;
- individual Groq calls use `LLM_TIMEOUT_SECONDS`, defaulting to 10 seconds.

## Catalog Grounding

Recommendations are generated by Groq from retrieved catalog context. The API then resolves every returned item against `shl_product_catalog.json`; items that are not present in the catalog are skipped. Test type codes are derived from the catalog `keys` field:

- `K`: Knowledge & Skills
- `P`: Personality & Behavior
- `A`: Ability & Aptitude
- `S`: Simulations
- `B`: Biodata & Situational Judgment
- `C`: Competencies
- `D`: Development & 360

## Evaluation

I replayed the 10 provided sample conversations as stateless histories. The implementation covers the required patterns: senior leadership, Rust/networking, contact center, graduate finance, sales audit, industrial safety, healthcare/HIPAA, admin Excel/Word, full-stack engineering, and graduate trainee batteries.

## What Did Not Work

The earlier app mixed a terminal `input()` loop with FastAPI startup. That caused `uvicorn main:app` to prompt for `HR says:` during import. The earlier `/chat` also accepted a single `message` field instead of the assignment's full `messages` history. The current version removes those issues and keeps `DB.py` unchanged.
